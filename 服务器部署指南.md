架构
Web服务器/反向代理 (Nginx): 作为面向公众的入口。它负责处理所有传入的HTTP/S请求，将访问前端应用的请求直接响应静态文件，并将所有API请求（如 /api/*）转发给后端的应用服务器。
应用服务器 (Gunicorn + Uvicorn): 在后台运行Python FastAPI应用。Gunicorn作为进程管理器，可以启动多个Uvicorn工作单元(worker)，从而利用多核CPU资源，提高并发处理能力。
服务守护进程 (Systemd): 作为操作系统级别的服务管理器，确保后端应用能够在服务器启动时自动运行，并在意外崩溃后自动重启，是保障服务稳定性的关键。

技术部署要点总结
1. 环境准备与源码管理 (部署的基石)
服务器基础软件栈: 部署目标为Linux服务器 (推荐Ubuntu)，必须安装 Python 3.11, Node.js, Nginx 和项目指定的包管理器 uv。
环境变量管理 (.env): 这是对话中提出的第一个关键优化。严禁将API密钥、数据库地址等敏感信息硬编码在 config.py 等源代码文件中。最佳实践是使用 .env 文件在服务器上存储这些信息，并通过 python-dotenv 库在应用启动时加载到环境变量中。
版本控制忽略 (.gitignore): 必须配置 .gitignore 文件，以排除所有不应提交到代码仓库的内容，这包括：
敏感文件: .env（目前API硬编码在config.py）
依赖目录: node_modules, .venv, ot_env
构建产物: dist, *.egg-info
系统/IDE配置: __pycache__, .vscode/
2. 后端部署 (FastAPI) - 核心难点与解决方案
独特的“双虚拟环境”策略: 这是项目最显著的技术特点。为了解决 opentrons 库的依赖冲突，部署时需要创建两个独立的Python虚拟环境：
.venv: 用于主应用及其大部分依赖。
ot_env: 一个隔离环境，专门用于安装和运行 opentrons。
依赖锁定与安装: 为保证环境的可复现性，应使用 uv.lock 文件来锁定所有Python依赖的精确版本。在服务器上，使用 uv pip sync 命令可以快速、准确地根据锁文件安装依赖。
持久化运行 (systemd): 为了让后端服务稳定地在后台运行，需要为其创建一个 systemd 服务单元。配置文件 (otcode-api.service) 需明确指定：
User: 运行服务的Linux用户。
WorkingDirectory: 项目的根目录。
ExecStart: 启动应用的命令。生产环境推荐使用gunicorn启动，而不是uvicorn单进程。例如： gunicorn -w 9 -k uvicorn.workers.UvicornWorker backend.api_server:app。
安全配置 (CORS): 必须在FastAPI应用中收紧CORS（跨源资源共享）策略，将 allow_origins 从 ["*"] 修改为仅允许前端应用的域名，防止其他恶意网站的非法请求。
3. 前端部署 (React) - 标准流程
API地址配置: 在构建前，必须修改前端代码中请求后端的API地址，将其从本地开发环境的 http://localhost:8000 改为生产服务器的域名（例如 https://your_domain.com/api）。
构建静态文件: 在服务器上进入前端代码目录，执行 npm install 和 npm run build。此操作会将整个React应用编译打包成优化过的HTML, CSS, 和 JavaScript 静态文件，通常存放在 dist 目录中。
4. Nginx配置 - 串联前后端的桥梁
Nginx在此架构中扮演两个核心角色：
静态文件服务器: 配置 Nginx 直接托管前端 dist 目录下的静态文件。
反向代理: 配置 location /api 块，将所有匹配/api前缀的请求无缝转发到本地运行的后端Gunicorn服务（http://localhost:8000）。这有效地将前后端整合在同一个域名下。
启用HTTPS: 在生产环境中，必须使用HTTPS加密所有通信。对话中提到了使用 certbot 和 Let's Encrypt 为域名免费申请和自动配置SSL证书，这是保障数据安全的基础操作。
5. 进一步提高(可选项。从“能用”到“好用”)
结构化日志 (logging): 用Python的 logging 模块替代 print() 语句。这能实现日志分级、输出到文件、自动轮转等功能，是排查线上问题的生命线。
容器化 (Docker): 这是解决“环境不一致”问题的终极方案。通过为前后端分别编写 Dockerfile，并将整个应用栈（后端、前端Nginx）用 docker-compose.yml 进行编排，可以实现一键式、可复现、与宿主机隔离的部署。这极大地简化了环境配置的复杂性，特别是本项目中“双虚拟环境”的特殊需求。
CI/CD (持续集成/持续部署): 建议使用GitHub Actions等工具建立自动化流水线，实现代码提交后自动运行测试、构建和部署，提升开发效率和部署的可靠性。
